# 架构设计规约：v5.0 实机安全自适应期 (Real-Device Safe Residual Adaptation)

**前置依赖 / 当前状态说明**

> v5.0 是真实设备 / HIL / ground test 部署阶段，不是当前 debug 目标。必须在 v3.0 robust residual controller 稳定后才进入。

v5.0 进入前必须满足：

1. **v3.0 robust residual controller 已稳定**：在 domain randomization、伺服延迟、外扰力矩下，residual actor 已通过全部验收标准。
2. **如果使用 v4.0 Token/Attention**：必须先通过仿真和 HIL 验证，确认 Attention-based residual actor 在随机化环境下仍稳定。
3. **critic ensemble / uncertainty gate 有效**：ensemble 在 OOD 状态下能可靠检测 uncertainty 升高。
4. **safety shield 在仿真中已验证**：gimbal / wheel / momentum safety 机制在各种极端条件下已验证。
5. **实机不允许普通 TD3 actor 直接输出完整动作并自由探索**：这是硬性安全约束。

**风险说明**：实机部署涉及硬件安全、不可逆操作、有限数据。任何 actor 输出异常都可能导致硬件损坏。residual online adaptation 必须在严格安全约束下进行。

---

**1. 核心目标**

- 将仿真中训练好的 robust baseline 部署到真实设备；
- 通过小 residual online adaptation 补偿仿真-现实差距；
- 保持硬件安全、可回滚、可审计。

---

**2. 网络结构**

```
a_real = safety_filter(a_base(s) + α_real Δπθ(s))
```

其中：
- **a_base(s)**：v3/v4 得到的 robust baseline；
- **Δπθ(s)**：实机上允许继续学习的小 residual；
- **α_real**：非常小，例如 0.01 / 0.02 / 0.05；
- **safety_filter**：硬件安全层，对动作进行最终限幅和安全校验。

**实机第一阶段不建议直接使用 null-space actor。**

只有在真实控制分配矩阵和参数辨识可靠后，才考虑：

```
a_real = safety_filter(a_base(s) + α_real N_real(s) zθ(s))
```

其中 N_real(s) 基于真实参数辨识得到的 control allocation matrix。

---

**3. 实机训练流程**

### Phase 5-a：Baseline 数据收集

- 只运行 baseline `a = a_base(s)`，不更新 actor；
- 收集真实设备的状态-动作-下一状态数据；
- 目标是建立真实数据分布的 baseline estimate；
- 收集过程中全程监控 safety shield，记录任何触发事件。

### Phase 5-b：Critic warmup / calibration

- 用真实数据 warmup / calibration critic；
- critic ensemble 使用真实数据同步更新；
- 进行 critic audit：
  - Q 面是否合理；
  - ensemble disagreement 是否在可接受范围；
  - critic 是否对真实设备的 bad states 有正确的价值估计。

### Phase 5-c：Residual actor 小步更新

- **只在 critic audit 通过后**，才允许 residual actor 更新；
- α_real 保持极小（0.01 / 0.02）；
- 每次更新步数有限（例如 50~100 步）；
- 更新后立即进行 rollout eval：
  - 在安全区域内测试；
  - 对比 baseline 表现；
  - 检查动作质量指标。

### Phase 5-d：Rollout gate / Safety gate 判断

- 每次更新后通过 rollout gate / safety gate 判断是否保留；
- **如果性能变差**：rollback 到上一个 checkpoint；
- **如果触发安全边界**：fallback 到 baseline，停止 residual actor 更新，重新评估；
- **如果 safety shield 频繁触发**：降低 α_real 或暂停更新。

---

**4. Safety shield**

实机 safety shield 包含以下层次：

* **动作层安全**：
  - action limit：动作幅值硬限幅；
  - gimbal rate limit：框架角速度不超过伺服电机物理极限。

* **执行器层安全**：
  - wheel speed limit：飞轮转速不超过物理极限（绝对上限）；
  - wheel acceleration limit：飞轮加速度不超过驱动能力；
  - momentum health：系统总动量不超过安全阈值。

* **姿态层安全**：
  - 姿态误差边界：姿态误差超过阈值时触发紧急保护；
  - 角速度边界：角速度超过阈值时触发紧急保护。

* **硬件层安全**：
  - 电流限制；
  - 温度监控；
  - 振动监控；
  - 其他硬件安全指标。

* **应急机制**：
  - emergency fallback：紧急情况下直接切换到 baseline 或安全模式；
  - rollback checkpoint：每次更新前保存 checkpoint，可回滚到任意历史版本。

---

**5. Critic / ensemble 用途**

* **实机 ensemble critic 主要是 safety gate，不是探索发动机**：
  - critic disagreement 大时，不允许 actor 更新；
  - 不允许因为某一个 critic 给高 Q 就执行激进行为；
  - 只有 Q 稳定、uncertainty 低、rollout 不劣于 baseline，才允许保留更新。

* **Uncertainty gate**：
  - 当 ensemble disagreement 超过阈值时，标记当前状态为 OOD；
  - OOD 状态下 residual actor 输出应被进一步压缩（降低有效 α）；
  - 严重 OOD 时直接 fallback 到 baseline。

* **Critic 不用于鼓励探索**：
  - 实机上 critic 的唯一安全用途是阻止不安全的 actor 更新；
  - 不使用 Q 值作为探索信号。
  - critic 主要用于 update gate、uncertainty gate 和风险评估，而不是主动驱动实机探索。

---

**6. 验收标准**

* **实机 residual adaptation 不劣于 baseline**：
  - 固定角度 / 固定初始状态下的姿态恢复精度不低于 baseline；
  - 不因为 residual online adaptation 导致性能退化。

* **Safety shield 不频繁触发**：
  - 频繁触发表明 residual actor 探索了不安全区域；
  - 需要调整 α_real 或增加安全约束。

* **没有动作饱和**：
  - 动作不应长时间停留在 ±1 边界；
  - 动作饱和会导致控制失效和硬件过载。

* **没有高频震荡**：
  - 动作不应出现高频振荡；
  - 高频震荡会导致执行器磨损和控制品质下降。

* **不出现不可恢复姿态偏差**：
  - 在任何测试场景下，姿态偏差应能被 recovery 到安全范围；
  - 不允许出现姿态发散。

* **可回滚**：
  - 所有更新过程可回滚到最近一个或多个已验证安全 checkpoint；
  - rollback 机制必须在实际部署前验证。

* **所有训练/更新过程可审计**：
  - 记录每次更新的 critic audit 结果；
  - 记录每次 rollout gate / safety gate 的判断依据；
  - 记录所有 safety shield 触发事件；
  - 记录所有 rollback / fallback 事件。
